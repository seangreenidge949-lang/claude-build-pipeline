# 设计稿与产品文档的关系

## 元素有无 vs 位置样式

**铁律**：设计稿中 UI 元素的「有与无」必须严格 follow 产品文档，一个都不能缺。元素的「位置和样式」以设计审美为准，可以自由发挥。

**Why:** 产品文档定义了功能完整性（what），设计稿决定视觉表达（how）。缺元素=缺功能，位置不同=设计选择。前者是 bug，后者是创意。

**How to apply:**
1. 设计每个页面前，先从产品文档中提取该页面的所有 UI 元素清单（按钮、FAB、图标、标签、输入框等）
2. 设计完成后，逐项核对元素清单，确认每个元素都存在于设计中
3. 元素的具体位置（导航栏 vs 内容区 vs 浮动层）、大小、颜色由设计方向决定
4. 发现产品文档中的元素在设计稿中缺失 = must_fix，不可跳过

## 曾经犯过的错

2026-03-27 AI 会议记录助手设计中遗漏了：
- 追问 FAB（💬）：产品文档明确要求纪要详情页右下角 FAB
- TL;DR 复制按钮：产品文档明确要求卡片右下角复制图标
- 语音上传图标（🎙️）：产品文档明确要求导航栏右侧

根因：把每个页面当独立截图任务做，没有对照产品文档逐项检查元素完整性。

## 不可多——禁止装饰性添加

**铁律**：不得基于「设计常识」自行添加产品文档中不存在的 UI 元素。产品文档没有的，设计稿就不应该有。

**Why:** designer agent 会基于移动端设计规范（如「大多数 App 需要 Tab Bar」）自行添加产品文档中不存在的元素。这和缺元素是同一问题的两面——多了也是 bug。

**How to apply:**
1. 设计完每个页面后，除了检查「有没有缺」，还要检查「有没有多」
2. 页面上的每个可交互元素（按钮/图标/Tab/FAB）都必须能在产品文档中找到对应描述
3. 找不到对应描述的元素 = 装饰性添加 = 必须删除

**曾经犯过的错：**
- 01-首页添加了底部 Tab Bar（首页/搜索/设置）：产品文档没有 Tab Bar，只有 FAB(+)
- 01-首页添加了搜索图标：页面已有搜索栏，图标和搜索栏功能重复
- 02-输入原文同时有导航栏语音图标 + 底部语音上传按钮：同一功能两个入口，产品文档只定义了导航栏图标

## 优先级铁律

**产品文档 > 设计规范 > 设计直觉**

- 产品文档是具体决策（「这个产品要什么」）
- 设计规范是通用建议（「大多数产品应该怎样」）
- 设计直觉是经验推断（「我觉得应该有」）

冲突时，上位覆盖下位。设计规范说 Tab Bar 是标配，但产品文档没有 Tab Bar → 不加。

## 回滚/修改后的完整性检查

回滚操作的目标不是「恢复到上一个状态」，而是「恢复到产品文档定义的状态」。

**How to apply:**
- 回滚完成后，MUST 对照产品文档逐项检查修改过的页面
- 检查项：元素不缺（产品文档有的都在）+ 元素不多（产品文档没有的都不在）
- 不做这个检查的回滚 = 盲目回滚，可能引入新问题
