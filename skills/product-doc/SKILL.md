# App 功能设计文档

> 定位：写清楚"做什么 + 业务逻辑怎么转"，让 AI 读完能直接写代码。
> 只写 AI 无法自行推导的信息。通用交互模式（按钮三态、网络错误处理、空状态、退出确认等）不在文档中穷举，由 coding 阶段的 UI 自检 checklist 保障。

## 什么该写 / 什么不该写

| 该写（AI 猜不到） | 不该写（AI 默认行为就是对的） |
|---|---|
| 业务规则：当X→系统Y→结果Z | 按钮 loading/disabled 状态（通用模式） |
| 枚举值和映射表（如状态→颜色） | 网络错误→显示重试（标准异常处理） |
| 非目标边界 | 返回时弹"放弃编辑？"（通用 UX） |
| AI prompt + 输出 schema | Given-When-Then 验收标准（业务规则已覆盖） |
| 反直觉的交互决策 | 布局概要（除非有非常规布局） |
| 技术约束（API/栈/超时） | 闭环三要素表（业务规则表已覆盖跳转逻辑） |

**一句话原则：如果 AI 的默认行为就是对的，不写；如果 AI 的默认行为会错，必须写。**

## 产出方式

按 Step 分段写入文件。每段用 Write/Edit 工具直接写入目标文件。

## 流程

> **每次运行开始前**：`Read references/reflection.md`，加载历史沉淀的已知约束和写作模式，避免重复踩坑。

### Step 1: 确认产品骨架（用户必须确认才进入 Step 2）

> 这是唯一需要用户深度参与的环节。目标：让用户看到产品的完整形态并确认"对，就是这样"。

#### 1a: 收集核心输入

先检查用户是否已提供需求简报或类似文档（功能清单+页面结构+主路径）。
- **已有**：直接复用，不重复问。补充缺失信息即可。
- **没有**：与用户共创以下四件事（1-2 轮对话完成）：
  1. **产品名称 + 一句话定位 + 目标平台**
  2. **功能清单**：这个 App 要做哪些事（**核心功能至少 4 个**，每个核心功能必须有对应主路径描述；用户给的少于 4 个时，主动追问直到补齐）
  3. **不做什么**：主动追问非目标边界
  4. **外部 API / 技术约束**（有 AI 功能或第三方 API 时必问）：
     - 用什么 AI proxy / API？有没有特殊的 model name 格式？（对照 `references/reflection.md` 预判，用户确认后直接写入文档）
     - 有没有已知的 SDK 版本限制或 bug？（如 iOS 最低版本）
     - 用户没主动说的，用 reflection 中的已知值填入并告知用户"已使用上次的配置"

#### 1b: 输出骨架文档，等用户确认

基于收集到的信息，输出**产品骨架**（直接写入目标文件）：

```markdown
# <产品名> — 产品设计文档
> 平台：<...> | 日期：<...>

## 功能列表
| 功能 | 一句话描述 | 实现优先级 |
|------|-----------|-----------|
| 文字输入 | 大文本框粘贴会议记录 | P0 |
| AI 整理纪要 | 输出结构化纪要+氛围标签 | P0 |
| ... | ... | ... |

## 页面结构
App
├── 首页（历史列表）
│   └── FAB → 新建记录页
├── 新建记录页
│   └── 「开始整理」→ AI 处理页
├── ...

## 功能路径
每个功能的主路径（用户做什么 → 系统做什么 → 用户看到什么）：

- **文字输入**：用户粘贴文本 → 点击「开始整理」→ 跳转 AI 处理页
- **AI 整理纪要**：系统调用 AI → 返回结构化 JSON → 自动跳转纪要详情页
- **搜索**：用户输入关键词 → 实时过滤历史列表 → 显示匹配结果
- ...

## ⛔ 非目标
- 实时语音转写 —— 原因：...
- ...
```

#### 1c: 生成可交互 HTML 原型（与骨架同步输出）

骨架文档写入后，同级目录生成 `{产品名}-prototype.html`：

**HTML 结构规范（单文件，不依赖 CDN）：**
- **左侧导航**：可折叠的页面树，层级对应骨架的「页面结构」树
- **中间主区**：375px 宽的手机帧，展示当前选中页面的模拟内容
- **每个页面按功能语义生成内容**（不是空框）：
  - 列表类 → 2-3 条模拟卡片 + 操作按钮（FAB / 顶部按钮）
  - 表单类 → 对应输入框 + 提交/取消按钮
  - 详情类 → 标题区 + 内容块 + 底部操作按钮
  - AI 处理类 → loading 动画 + 进度提示文案
- **跳转按钮严格按「功能路径」连线**：Step 1 主路径中的"→"操作全部对应真实可点击按钮，点击切换页面
- 页面切换用 JavaScript 控制 display，无需服务器
- 颜色用功能优先级标注：P0 页面标题用蓝色，P1 用橙色，P2 用灰色
- **右侧反馈侧边栏（默认隐藏，按需呼出）**：
  - 手机帧右边缘固定一个竖排 tab「反馈 ›」，点击侧边栏从右侧滑入，再点收起
  - 侧边栏内容随当前页面自动切换，只展示**当前页**的内容：
    - 「页面内容」区：列出该页所有 UI 元素（如"搜索栏""卡片列表""FAB 按钮"），每条配一个文本输入框
    - 「出向跳转」区：列出该页所有跳转操作（如"FAB → 新建记录页"），每条配一个文本输入框
  - 底部全局「导出全部反馈」按钮：将所有填写过的条目格式化后复制到剪贴板，格式为：
    ```
    页面确认反馈：
    ✅ 首页 - 通过
    ❌ 新建记录页
      页面内容 / 大文本框：应该有标题输入框
      出向跳转 / 开始整理 → AI处理页：失败时应留在当前页不跳转
    全局备注：（如有）
    ```
  - 没有填写任何内容的页面在导出时标记为 ✅ 通过，有内容的标记为 ❌

写入完成后立即执行 `open {文件绝对路径}` 在浏览器中打开原型，并告知用户：
`💡 原型已在浏览器打开。在原型里点击体验跳转，对每个页面的调整意见可点右边缘「反馈 ›」填写，填完后点「导出全部反馈」，把复制的内容粘贴回来即可。`

**MUST 等用户反馈后才进入 Step 2。** 不再用 AskUserQuestion，等用户粘贴反馈内容或直接说"没问题"。

#### 骨架的四个交叉检查（输出前自检）

输出骨架前，内部检查：
1. 功能列表中的**每个功能**在页面结构树中都有对应页面
2. 页面结构树中的**每个页面**都能追溯到某个功能
3. **每个功能**都有主路径描述（不能遗漏任何一个）
4. **核心功能数量 ≥ 4**，且每个核心功能都有主路径；不足时退回 1a 补齐后再输出

### Step 2: AI 推导完整文档（不需要用户逐段确认）

> 用户已确认骨架，AI 基于骨架自动展开所有细节。遇到业务逻辑不确定时才暂停追问。

#### 2a: Why + Who

**读取**：`Read references/01-写作规范.md`

在骨架文档的功能列表前插入：
- **Why**：背景 + 目标（可量化）
- **Who**：目标用户 + 至少1个完整使用场景（写到"用户看到什么结果"）

#### 2b: 逐功能展开（核心章节）

**读取**：`Read references/02-功能设计模板.md`

**所有功能同等展开**——优先级只影响实现顺序，不影响文档详细程度。

每个功能按模板展开：
1. **概述**：一句话
2. **入口与退出**：从哪进 + 从哪回
3. **业务规则**：从 Step 1 的主路径出发，用"当X → 系统Y → 结果Z"表格补全所有条件分支（正常+异常+边界）
4. **反直觉交互**（仅在有时）：AI 默认行为会出错的交互决策
5. **数据模型**：`Read references/03-数据模型规范.md`，用 code block 定义实体

若涉及 AI 功能，额外写入：
- **Prompt 模板**：完整文本，包含所有变量占位符
- **输出 JSON schema**：所有字段名、类型、值域范围（如 `frequency_days: 整数, 1-30`）
- **解析防御规则**（必写，参照 `references/reflection.md` 的已知陷阱）：
  - 枚举字段的异常值降级规则（AI 返回枚举外的值时统一降级为哪个值，不能让未定义值进 UI）
  - 特殊字符处理（string value 中是否可能出现未转义字符，如何清洗）
  - `max_tokens` 估算依据（防止 JSON 末尾截断）
- **fallback 策略**：API 失败/超时的降级逻辑

#### 2c: 技术约束

以下情况**必须写**（不可跳过）：
- 有 AI 功能 → 写 API proxy 信息、model name 格式、timeout 值、已知版本 bug
- 有平台版本限制 → 写最低版本要求及原因
- 有非标准第三方依赖或特殊初始化 → 写说明

以上均无时跳过。

### Step 3: 独立审查 + 交付

> 自我校验存在作者认知偏差，此步骤改为派发独立 subagent 审查。

🤖 调用 subagent: plan-reviewer

以下参数传入：
- `plan_path`: 本次写入的产品文档完整路径
- `spec_path`: 若 Step 1 有用户提供的原始需求文档，填入其路径；否则为空
- `other_docs`: 空
- `mode`: "standard"
- `known_pitfalls`: 空

**等待审查结果后：**

- 裁决为 **🚫 BLOCK**：将所有 CRITICAL / MAJOR issue 修复到文档中，然后重新派发 plan-reviewer 再审一轮，直到裁决不为 BLOCK。
- 裁决为 **⚠️ CONDITIONAL PASS**：将 MAJOR issue 修复到文档中，MINOR 问题告知用户并由用户决定是否修复。
- 裁决为 **✅ PASS**：直接交付。

修复完成后告知用户文档路径，展示目录结构摘要。

#### 3b: 审查问题回写（每次审查后必须执行）

无论裁决结果如何，只要 reviewer 发现了任何 issue（CRITICAL / MAJOR / MINOR），必须执行以下两步回写：

**回写 1 — 积累 known_pitfalls（`references/05-reviewer-pitfalls.md`）**

将本次所有 issue 追加到 `references/05-reviewer-pitfalls.md`，格式如下。文件不存在时创建。

```
## [YYYY-MM-DD] <产品名>
- 🔴 [issue 简述]
- 🟠 [issue 简述]
- 🟡 [issue 简述]
```

此文件在下次调用 plan-reviewer 时作为 `known_pitfalls` 传入。

**回写 2 — 写作规范自升级（`references/reflection.md`）**

对每一个 issue，追问：**"这个问题，我在写作阶段本可以避免吗？如果可以，我应该在哪个环节做什么？"**

- 如果答案是"是"（写作层可预防）：将对应的写作规则追加到 `reflection.md` 的合适章节（同 Step 4 的写入规则：带 `[YYYY-MM]` 前缀，只写可泛化的规则，不重复已有内容）。
- 如果答案是"否"（纯粹的产品决策问题，写作层无法预防）：跳过，不写入。

回写完成后，向用户输出一条总结：

```
📊 审查问题回写完成
  已转化为写作规则：X 条（写入 reflection.md）
  非通用问题跳过：Y 条（原因：[简述各条为何不可泛化]）
```

**目标**：每次 reviewer 发现的问题，都转化为 product-doc 下次写作时的防御规则。reviewer 发现的问题越来越少，说明写作质量在提升。

### Step 4: 反思 & 沉淀（强制，不可跳过）

> ⚠️ 这是 skill 的强制完成条件，不是可选步骤。未执行此步骤 = skill 未完成。
> 即使本次无新发现，也必须写一行空记录，证明反思已执行。

> 目标：把这次运行中发现的新知识写回 `references/reflection.md`，让下次运行更好。

用 Edit 工具追加到对应章节末尾。**只写新发现，不重复已有内容。每条必须带 `[YYYY-MM]` 日期前缀。**

**写入前先判断可泛化性**：这个发现是"这次特有"还是"以后大概率还会遇到"？只有后者才写入，前者记在会话里即可。

反思四个问题：

**Q1：这次追问出了哪些用户没主动说的约束？**
→ 如果是 API/技术约束（proxy 格式、timeout、版本 bug），追加到 `## AI API / 外部接口约束` 或 `## 技术栈约束`

**Q2：有没有遇到"AI 默认行为会猜错"的场景，且 reflection 里没有记录的？**
→ 追加到 `## 文档写作模式 > 什么情况下 AI 会猜错`

**Q3：有没有某类问题在骨架阶段特别容易被遗漏，值得加入"高价值追问"清单？**
→ 追加到 `## 文档写作模式 > 高价值追问`

**Q4：这次有没有某个章节结构或写法让文档特别清晰，值得固化为标准模式？**
→ 追加到 `## 成功模式`

如果四个问题的答案都是"无新内容"，写一行：`<!-- [YYYY-MM] 本次运行无新沉淀 -->`。

**定期维护（文件超过 150 行时触发）**：检查有无可删除的条目——技术版本已升级的约束、已被写作规范吸收的条目。找到候选条目后，告知用户"以下条目可能已过时，是否删除：…"，等用户确认后再删除。
