## AI API / 外部接口约束  [触发时机: Step 1 追问 / Step 2 技术约束章节]

### mify proxy（用户惯用 AI proxy）

- [2026-01] **model name 必须带 `ppio/pa/` 前缀**，如 `ppio/pa/claude-sonnet-4-6`。缺少前缀返回 400 且**静默失败**——代码层必须打印完整 HTTP 错误体，否则极难调试
- [2026-01] **Vision API timeout 应设为 60s**，不是默认的 10s。图片 base64 + 结构化 JSON 输出实际耗时 30-50s，10s 会大量超时
- [2026-01] **max_tokens 建议 1500**：结构化中文 JSON 约 800-1200 token，需要 buffer 防止末尾截断导致 JSON 解析失败

### 中文 LLM JSON 解析陷阱（通用，所有中文 LLM 均适用）

- [2026-01] **未转义双引号**：string value 中可能出现未转义的 ASCII `"`（如 `用"阳光"补充`），解析前必须对 value 内的 `"` 做清洗（替换为 `'`）
- [2026-01] **枚举异常值**：AI 可能返回枚举外的值（如 `"very_high"` 而非 `"high"`，或字段缺失）。文档中必须定义降级规则（如"统一降级为 `low`"），不能让未定义值进入 UI 逻辑

## 技术栈约束  [触发时机: Step 1 追问 / Step 2 技术约束章节]

### iOS / SwiftData

- [2026-01] **iOS 17.0 SwiftData 有 bug**，最低版本应设为 17.2+，文档中技术约束必须注明
- [2026-01] **照片存储**：不要把图片 Binary Data 存入 SwiftData（严重影响查询性能），应存文件系统路径。路径格式建议：`{entityType}/{entityId}/{timestamp}.jpg`，存相对于 App Documents 目录的相对路径，不存绝对路径（App 重装后绝对路径会变）

## 文档写作模式  [触发时机: Step 2 业务规则展开时对照检查]

### 什么情况下 AI 会猜错

- [2026-01] **重复数据是否去重**：AI 默认会加去重逻辑。如果业务允许重复（如多盆同品种植物），必须在业务规则中明确写"允许重复，不做校验"
- [2026-01] **关联记录更新策略**：重新识别/编辑时，哪些字段自动更新、哪些由用户确认、哪些不变——AI 会默认全量覆盖，实际上应该分字段处理
- [2026-01] **提醒/任务频率是否随建议更新**：AI 默认会同步更新。如果用户已手动调整过频率，不应自动覆盖——必须明确写出来
- [2026-01] **用户手动填写的字段在 AI 重新处理时是否覆盖**：AI 默认全量更新所有字段；如果有字段是"用户主观输入"（如备注、评分），必须显式写"AI 重新分析不覆盖该字段"
- [2026-01] **多标签/多条件筛选的 AND vs OR 逻辑**：AI 默认 AND（更精确），但用户日常习惯往往是 OR（"看这几类"）；凡有多标签筛选，必须显式说明取并集还是交集
- [2026-01] **用户偏好设置是否跨 Session 保留**：AI 默认持久化所有用户设置；但某些筛选/临时状态持久化后会让用户误以为"内容消失"——必须显式写是否持久化

### 高价值追问（骨架阶段容易遗漏）  [触发时机: Step 1 收集核心输入]

- [2026-01] **AI proxy / 认证方式**：用户不会主动说，但影响整个技术约束章节
- [2026-01] **离线能力边界**：哪些功能断网可用、哪些不行——影响数据模型设计（本地存储 vs 只缓存）
- [2026-01] **同一实体多次操作的更新策略**：如"再次识别同一植物后，哪些字段更新"——容易被遗漏，后期改代价高
- [2026-01] **删除时的级联行为**：凡是实体关联了外部资源（图片、文件、远程数据），必须明确"删除时是否同步删除关联资源"——AI 默认只删 DB 记录

## 文档写作模式  [触发时机: Step 2 业务规则展开时对照检查]

### 高价值追问（骨架阶段容易遗漏）  [触发时机: Step 1 收集核心输入]

- [2026-04] **用户限额的边界行为**：凡是有"每用户最多N条"的规则，必须追问：达到上限时点击入口按钮的前端行为是什么？是进入表单后提交才报错，还是入口处直接拦截？
- [2026-04] **分享/邀请码的使用方式**：一次性还是多人可用？每人使用后是否失效？设计增长机制时必须明确，否则数据模型无法确定
- [2026-04] **同一内容在不同场景下的组件差异**：如付费引导弹窗在报告/企业/需求三个场景下是否相同——场景差异必须明确，不能用"参考其他场景"

### 什么情况下 AI 会猜错

- [2026-04] **付费内容的 API 权限区分**：AI 默认会用 403 区分权限，但"未登录"和"非会员"在前端要展示不同弹窗。正确做法是未登录返回 401，非会员返回 200+null+membershipRequired:true，必须在技术约束中明确
- [2026-04] **多套枚举是否对齐**：同一概念（行业/地区/状态）如果在多个模块各自定义枚举，AI 会产生不一致实现。写文档时必须统一枚举，或明确说明差异理由

## 成功模式  [触发时机: Step 2 展开时参考结构]

> 记录"这样写用户特别满意"的结构和写法，固化为可复用模式。

- [2026-04] **分享/增长机制设计**：分享解锁的完整机制用两张表描述最清晰：① 付费引导弹窗规则表（列出各触发场景的行为）；② 分享解锁机制表（用步骤列表描述完整链路）。两张表配合比混写在一起更易理解

## [2026-04] 全球产业链接智库 App — 首次演示反馈

### 本次发现的产品文档漏洞

**1. 功能入口重叠未定义去重规则**
- 问题：「收藏报告」统计数字 vs「我的收藏」功能入口并存，用户反馈重叠
- 根因：产品文档只定义了各入口的功能，没有定义多入口时的取舍规则
- 修复后规则：已在文档「功能交互通用规则」章节中明确

**2. 带交互暗示的 UI 元素行为未定义**
- 问题：章节目录有「›」箭头但产品文档未说明是否可点，研发实现为纯装饰
- 根因：产品文档只描述「展示章节目录」，没有说明点击行为
- 修复后规则：UI 元素的点击行为必须显式声明，未声明默认无交互

**3. 权限变化后的刷新时机未定义**
- 问题：会员开通后当前报告详情页不刷新，用户需手动重进
- 根因：产品文档只说「会员可解锁」，没有定义解锁后的 UI 更新时机
- 修复后规则：权限变更后的刷新策略必须在文档中明确

## [2026-04] 全球产业链接智库 App — 第二次演示反馈

### 统计数字硬编码问题
- 问题：「我的」页面收藏报告/浏览企业/发布需求三个统计显示固定数值 12/38/3
- 根因：产品文档说明了「展示统计数字」，但没有说明数据来源（用户实际数据 vs 演示数据 vs 写死）
- 修复后规则：任何展示「用户行为数据」的 UI 组件，产品文档必须明确：数据来源（API 字段名）、空值处理（0 还是 -）、更新时机（实时/每次进入页面）。未明确则默认研发写死演示数据，产品不得事后质疑。

### 功能重叠未定义主入口
- 问题：报告详情页有收藏图标，个人主页统计数字也显示收藏数量，用户不知道哪个是主入口
- 根因：产品文档分别定义了「报告详情-收藏功能」和「个人主页-统计展示」，但没有说明当两者在同一使用场景出现时谁优先
- 修复后规则：同一功能在不同页面出现时，必须在文档中指定「主入口」（用户期望在哪里完成这个操作）和「辅助展示」（仅读取状态，不作为操作入口）

## [2026-04] 全球产业链接智库 App — 第三次演示反馈

### 数据一致性：子集 vs 全集混淆
- 问题：企业详情页「商业洞察」显示 3 条，点击「查看全部」后只显示 2 条
- 根因：产品文档定义了「企业详情-商业洞察预览（前N条）」和「全部商业洞察页」，但没有明确两者共享同一数据源，研发各自从不同地方取数
- 修复后规则：凡是「预览 + 查看全部」的组合，产品文档必须显式声明：全部页的数据是预览数据的完整超集，字段结构完全一致，不是另一个接口的数据

### 列表页数据范围：用户子集 vs 全量
- 问题：「我的」页面点击「收藏报告/浏览企业/需求浏览」进入的是全量列表，而非该用户实际操作过的内容
- 根因：产品文档只说「点击进入列表页」，没有说明列表数据范围是「当前用户行为产生的子集」还是「全平台数据」
- 修复后规则：从「我的/个人主页」模块跳转的任何列表页，产品文档必须明确数据范围：① 用户子集（只显示该用户操作过的）② 全量（全平台内容）。未明确则默认全量，产品不得事后质疑。

### 功能删除需明确影响分析
- 问题：研发执行「去掉冗余收藏按钮」时，将企业详情底部的收藏按钮一并删除，但该按钮是企业详情唯一的收藏入口
- 根因：产品文档对「收藏」入口的分布没有全局视图，只在各页面分散描述；研发执行去重指令时缺乏全局判断
- 修复后规则：产品文档必须为每个核心操作（收藏、分享、订阅等）维护「入口分布表」，列出所有页面中该功能的入口位置和交互方式。执行「去掉某个入口」的需求时，必须先对照入口分布表确认是否为唯一入口。

### 导航栏冗余功能图标
- 问题：企业列表页已有页内搜索功能，导航栏右上角又放了搜索图标，造成功能重复
- 根因：导航栏功能图标和页面内功能区没有联动检查，各自独立定义
- 修复后规则：凡是页面内已有某功能的操作区域（搜索框、筛选器等），产品文档必须明确声明导航栏右侧是否还需要同功能图标。默认规则：页面内已有操作区 → 导航栏不重复放图标。

## [2026-04] 全球产业链接智库 App — 第四次演示反馈

### 责任归属判断

**产品责任（产品文档漏写或写错）**：
1. **企业关注功能完全缺失**：产品文档只写了「报告收藏」，没有定义「企业关注」功能。研发无法自行推断，属于产品漏写。
2. **商业洞察详情页缺失**：产品文档第277行写「移除遮挡层，展开商业洞察内容」，但没有定义「商业洞察详情页」（每条洞察是一篇完整商业报道，可展开查看）。属于产品定义不完整。
3. **我的页面子页面定义不完整**：产品文档页面结构（第50行）写了「我的收藏页」和「我发布的需求页」，但没有写「关注企业页」，也没有明确这些是独立子页面而非跳转到全局 Tab。属于产品漏写。
4. **订阅管理页会员态展示未定义**：产品文档只定义了选套餐流程，没有定义「已是会员后订阅管理页应显示什么（权益+到期时间，不再显示套餐选择）」。属于产品漏写。
5. **全球热点横幅点击行为与代码不符**：产品文档第142行明确写了「点击热点词 → 以该关键词触发全局搜索」，但第203行又写「全球热点横幅为纯展示性组件，不支持点击跳转搜索」。产品文档自相矛盾，需明确统一。本次用户反馈是「去掉可点击符号」，以第203行为准，纯展示不可点。

**研发责任（产品写了但没有实现）**：
1. **报告详情页收藏功能缺失**：产品文档第196-197行明确写了收藏/取消收藏的完整业务规则，研发未实现。
2. **报告全文章节与目录无对应关系**：产品文档第200行写「章节目录完整展示」，第204行写「会员解锁后返回全文自动刷新」，但没有写「章节目录可展开」——实际应实现章节目录与正文一一对应、点击章节展开对应内容。这是研发理解不足，应主动追问。
3. **我的页面菜单项重复**：产品文档页面结构已定义「我的收藏页」「我发布的需求页」为独立子页面，研发实现为跳转到全局 Tab，属于理解偏差。
4. **企业库右上角搜索图标**：上一轮反思已写入规则，本轮仍未修复，属于研发遗漏。

### 新增写作规则

- [2026-04] **功能对称性检查**：凡是为某类内容定义了「收藏/关注」功能，必须同时检查其他同级内容模块是否需要相同功能（如报告可收藏→企业是否可关注）。未显式声明不支持视为漏写。
- [2026-04] **会员态页面双状态定义**：凡是有「开通会员」流程的页面，产品文档必须同时定义：① 非会员态展示什么；② 已是会员后该页面展示什么。只定义其中一种状态的文档不完整。
- [2026-04] **产品文档不写位置和样式**：产品文档只定义「什么功能、什么条件触发、系统做什么、用户看到什么结果」，UI 元素的位置（左上/右上/底部）和视觉样式（颜色/大小/形状）由设计稿决定，不写入产品文档。
- [2026-04] **「查看全部」必须定义子页面结构**：凡是列表预览区有「查看全部」入口，产品文档必须定义对应子页面的完整结构（数据范围、排序规则、每条记录的展开行为）。未定义子页面结构不得实现该入口。
- [2026-04] **用户操作结果必须定义持久化范围**：凡是用户对某条记录有操作行为（收藏、关注、预约、已读等），产品文档必须明确：① 该状态是否需要跨设备同步（→ 后端 API）；② 是否需要本机跨会话保留（→ 本地存储）；③ 还是仅当次会话临时（→ 前端 state）。未定义持久化范围，研发默认按「临时」处理，产品不得事后质疑。
