# 功能设计模板

> 每个功能按以下结构完整展开。优先级只影响实现顺序，不影响文档详细程度。

---

## 单功能模块结构

```markdown
### [功能名称]

#### 概述
一句话说明这个功能做什么。

#### 入口与退出
- **入口**：从哪里进入（页面/Tab/按钮/深链接）
- **退出**：怎么离开（返回上一页/关闭弹窗/下滑dismiss）

#### 业务规则

| 场景 | 触发条件 | 系统行为 | 用户看到的结果 |
|------|---------|---------|--------------|
| 正常提交 | 表单填写完整 + 点击提交 | 调用 POST /api/xxx | 跳转到详情页 + Toast"提交成功" |
| 字段缺失 | 必填字段为空 + 点击提交 | 前端拦截，不发请求 | 缺失字段标红 + 提示"请填写XXX" |
| 提交失败 | API 返回 5xx | 不跳转 | 页面顶部 Banner"提交失败，请重试" |

枚举值必须穷举，不能用"等"。
条件必须可写成 if 语句。

#### 反直觉交互（仅在有时写）

AI 默认行为会出错的交互决策。只写"意外"的部分：

示例：
- "编辑模式下，氛围标签不可编辑（由 AI 生成，保持客观性）"
- "删除操作不需要二次确认（列表项可撤销恢复）"
- "已完成项排到列表底部（不是保持原位）"

如果所有交互都符合直觉（按钮三态、网络重试、空状态等），此章节跳过。

#### 数据模型

用 code block 定义，字段名+类型+注释即可（参见 03-数据模型规范.md）：

```dart
class Order {
  String id;              // UUID, 主键
  String title;           // 订单标题, 必填
  Decimal amount;         // 金额（元）, 默认 0.00
  OrderStatus status;     // draft/paid/shipped/completed
  DateTime createdAt;     // UTC, 自动生成
}
```
```

---

## 写作要点

1. **业务规则表是核心产出**——这是开发实现的依据
2. **枚举穷举**：状态有几种、分类有几个，全部列出
3. **只写 AI 会猜错的**：通用模式（loading、error、empty）不用写，反直觉决策必须写

## 页面结构写法

用页面树表示导航关系：

```
App
├── 首页 Tab
│   ├── 推荐列表
│   └── 搜索入口 → 搜索结果页
├── 发现 Tab
│   └── 分类浏览 → 详情页
├── 我的 Tab
│   ├── 个人资料 → 编辑资料页
│   └── 设置 → 设置页
└── 全局
    └── 登录/注册流程（未登录时拦截）
```

每个页面标注：该页面属于哪个功能模块。
