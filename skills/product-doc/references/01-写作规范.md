# 写作规范

> 消灭模糊描述，让 AI 读完不需要猜。

---

## 禁止用语 → 精确替代

| 禁止 | 替代写法 |
|------|---------|
| "用户可以…" | "点击X → 系统执行Y → 用户看到Z" |
| "合适的提示" | 写出具体提示文案，如"网络连接失败，请检查网络后重试" |
| "酌情处理" | 给出明确规则，如"金额 > 1000 时需要二次确认" |
| "等" | 穷举所有情况，或写"仅包含以上 N 种" |
| "参考XX端" | 把逻辑完整写到本文档 |
| "合理的默认值" | 写出具体默认值 + 选择理由 |
| "同上" / "类似" | 精确引用章节名，或把逻辑复制过来 |
| "正常流程" | 画出完整流程，没有"正常"和"非正常"之分 |
| "友好的提示" | 写出具体文案 |

---

## 业务规则描述格式

用"触发 → 动作 → 结果"三段式，不用叙述体：

```
当[触发条件]时 → 系统[执行动作] → [用户看到的结果]
```

示例：
```
当用户连续输错密码 5 次时 → 系统锁定账号 30 秒 → 登录按钮变灰 + 显示倒计时"30秒后可重试"
```

条件判断必须可写成 if 语句。"用户活跃时" → "最近 7 天有过登录行为时"。

---

## 数值具体化

| 模糊 | 具体 |
|------|------|
| 约 3 秒 | 3 秒 |
| 比较大的间距 | 16px |
| 适当延迟 | 300ms |
| 用户活跃时 | 最近 7 天有过登录行为时 |

## 非目标写法

```
⛔ 本期不实现：
- 搜索建议/联想词功能 —— 原因：MVP 优先核心搜索，后续迭代
- 移动端响应式 —— 原因：初期只面向桌面端用户
- 多语言支持 —— 原因：当前只服务中文用户
```

每条必须有理由。没有理由的非目标 = 没有边界。

---

## 验收标准：Given-When-Then

每个功能的验收标准用 GWT 格式，覆盖正常 + 异常 + 边界：

```
Scenario: 正常搜索
  Given 用户在搜索框输入"耳机"
  When  点击搜索按钮
  Then  跳转到搜索结果页，显示匹配商品列表

Scenario: 空搜索
  Given 搜索框为空
  When  点击搜索按钮
  Then  搜索框边框变红，显示"请输入搜索内容"

Scenario: 无结果
  Given 搜索"xyzabc123"
  When  API 返回空数组
  Then  显示空状态页"没有找到相关商品"
```
