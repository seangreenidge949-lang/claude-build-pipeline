# 产品文档写作经验教训

> 来源：TV OS4 天气需求文档评审（2026-03-10），收到 33 条评论

## 评审问题分类

| 类别 | 占比 | 典型问题 |
|------|------|---------|
| 数据映射缺失 | 36% | 天气等级/文案/颜色只给示例不给全表；"同手机"未展开具体映射 |
| 技术可行性 | 24% | API 字段未确认就写入方案；系统能力未验证 |
| 交互逻辑歧义 | 18% | 遥控器焦点路径不完整；按键映射不明确 |
| 边界策略缺失 | 12% | 无网络/定位失败/数据过期等异常场景未覆盖 |
| 系统冲突 | 9% | 天气通知与已有系统通知机制的关系未说明 |

## 根因分析

1. **写了"是什么"但没写"怎么算"**：描述了功能表现，但缺少驱动这些表现的具体规则（映射表、阈值、公式）
2. **先写方案后验可行性**：假设 API 存在且字段满足需求，写完方案才发现接口限制
3. **"同手机"不是有效定义**：研发不应需要去查另一个端的代码来理解本端的需求
4. **交互描述停留在用户视角而非实现视角**：说"用户可以切换城市"但没说遥控器焦点怎么移动、按什么键触发

## TV/遥控器交互规范要点

TV 端需求文档必须额外关注：

### 焦点路径图
- **每个页面必须有焦点路径图**：标注默认焦点位置、方向键导航路径、焦点循环规则
- **跨区域导航**：当页面有多个功能区时，需明确区域间的焦点切换逻辑
- **焦点记忆**：返回页面时焦点是否恢复到上次位置

### 按键映射
- OK 键、返回键、菜单键的行为必须逐页面定义
- 长按行为（如果有）需单独说明
- 数字键快捷操作（如果有）需说明

### TV 特有约束
- 无触摸、无键盘输入 → 文本输入场景需特殊方案
- 观看距离远 → 字号/信息密度有硬性约束
- 常驻后台 → 数据刷新策略比手机更重要（定时刷新 vs 事件触发）

## 通用教训 vs TV 特定教训

### 通用（已固化到 Skill）
- 动态内容必须输出完整映射表，不能只给示例
- 技术依赖在方案撰写前确认，不能先写后验
- "同XX端"不是有效定义，必须把逻辑拉到本文档
- 埋点需明确触发时机和上报策略

### TV 特定（记录在此 Memory）
- 焦点路径图是 TV 需求的硬性交付物
- 遥控器按键映射需逐页面定义
- 常驻后台带来的数据刷新策略问题
- 大屏信息密度约束
