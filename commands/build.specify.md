# Stage 1: 需求明确 (Specify)

## Stage Contract
- **requires**: —（首个阶段，无前置依赖）
- **optional**: —
- **produces**: `.build/01-specify.md`
- **precondition**: 项目目录和 `.build/` 已创建

## 前置检查

运行前置检查脚本：
```bash
~/.claude/scripts/check-build-prerequisites.sh specify --json
```

解析 JSON 输出，获取 `project_root`、`available_docs` 等信息。
如果脚本返回非零退出码，停止执行并将 stderr 中的错误信息告知用户。
如果当前阶段已 completed，询问用户是否要重做。

## 用户输入

```text
$ARGUMENTS
```

## 本阶段的定位

> **specify 只做"把问题说清楚"，不做"想怎么解决"。**
>
> 这个阶段用户还没有调研基础。不应该要求确认功能范围、验收标准、技术选型——这些需要信息支撑的问题留给后续阶段。
> 也不应该选方向——方向是在看完 research、讨论完 value 之后才能定的。
>
> 这里只需要理解用户的**动机和问题**，产出足够 research 启动的信息。
> 终止条件不是"问够 N 个问题"，而是"有足够信息启动调研"。

## 执行流程

### Step 1: 识别项目类型和复杂度

根据用户描述，判断：

**文档类型**：
| 类型 | 触发信号 | 核心聚焦 |
|------|---------|----------|
| PRD | "需求文档"、"PRD"、"产品需求" | 做什么、为什么做 |
| 功能设计文档 | "功能设计"、"Feature Spec"、"详细设计" | 详细行为、边界情况 |
| 产品提案 | "产品方案"、"提案"、"立项" | 问题、机会、ROI |
| 完整产品 | "做一个App"、"做一个工具"、"开发" | 需求→设计→代码的全流程 |

**复杂度判断**（写入 `_manifest.json` 的 `complexity` 字段）：
| 复杂度 | 信号 |
|--------|------|
| small | 单一功能点、脚本/CLI/单文件 |
| medium | 2-5 个页面/模块、前后端 |
| large | 新品/新方向、跨多模块/多端 |

### Step 2: 理解动机——"为什么做"

> **核心问题不是"做给谁"，而是"为什么做"。**
> "为什么做"的答案自然包含了目标用户——"我自己有这个痛点"就是自用，"市场有需求"就有目标人群，"领导要求的"就有内部背景。

**第一个问题（必问）**：了解动机

从用户的 $ARGUMENTS 中判断动机类型，追问对应的深度信息：

| 动机类型 | 信号 | 追问方向 |
|---------|------|---------|
| **自己有痛点** | "我想做..."、"我需要..." | 这个痛点困扰你多久了？现在怎么凑合解决的？ |
| **市场有需求** | "用户需要..."、"市场上缺..." | 怎么发现的？有数据/案例吗？ |
| **公司/团队要求** | "领导说..."、"业务方需要..." | 背景是什么？有现成文档/讨论记录吗？ |
| **灵感/兴趣** | "我觉得..."、"看到XX想试试" | 什么触发了这个想法？你会是第一个用户吗？ |
| **不确定** | 描述模糊 | 先聊聊你遇到的问题/看到的机会？ |

**澄清原则**：

<HARD-GATE>
**一次只问一个问题**。禁止在同一条消息中抛出多个问题。
优先使用 AskUserQuestion 提供选项（含推荐标注），降低回答成本。
不在 specify 阶段讨论技术方案或功能设计——那是 plan 阶段的事。
不在 specify 阶段选方向——那是 value 阶段讨论后才定的。
</HARD-GATE>

### Step 3: 补充调研锚点

在理解动机后，检查 research 阶段还缺什么信息。以下信息是 research 必须的输入——缺哪个补哪个：

| 锚点 | 为什么需要 | 如何获取 |
|------|---------|---------|
| **问题描述** | 搜索关键词的核心来源 | 从 Step 2 的动机对话中提炼，向用户确认 |
| **产品形态** | 决定搜什么品类的竞品 | App/Web/CLI/硬件/小程序？如果动机对话中已提及则不重复问 |
| **已知约束** | 避免调研方向偏离 | 时间/平台/预算/技术限制？如果没有则标注"无已知约束" |
| **现有替代方案** | 竞品搜索的起点 | "你现在怎么解决这个问题的？"（如果 Step 2 已回答则不重复问） |

**不需要在这里确认的**（留给后续阶段）：
- 具体功能点 → plan 阶段
- 目标用户画像 → research 数据支撑后在 plan 定义
- 验收标准 → value 讨论后在 plan 定义
- 技术栈 → code-plan 阶段
- 方向选择 → value 讨论后定

**最少问题数保障**：Step 2（动机）+ Step 3（锚点）至少覆盖 4 个问题，即使前面的对话已自然覆盖了部分内容，也要确认性地过一遍。Step 4（灵魂追问）在此基础上追加。总问题数以"research 有足够信息启动"为终止条件。

### Step 4: 灵魂追问（根据项目特点选最有价值的）

> 灵魂追问是"搜索引擎不知道但对决策至关重要的上下文"。
> 这些答案会直接影响 research 的搜索方向和 value 的讨论重点。
> 不固定数量——如果 Step 2-3 的对话已经自然覆盖了某些问题，跳过。

从以下问题中选择**尚未被回答的**提问：

| 问题 | 适用场景 | 给后续阶段的价值 |
|------|---------|----------------|
| **你自己会用吗？用多频繁？** | 动机是"自己有痛点"时跳过（已知） | 判断"自己的痒"还是"别人的痒" |
| **做出来了谁最开心？** | 动机模糊或涉及多方时 | 验证目标用户一致性 |
| **做成什么样你觉得算成功？** | 几乎总值得问 | 成功标准锚点 → value 讨论和 plan 验收标准的种子 |
| **如果只能做一件事，你觉得最重要的是什么？** | 需求偏大或模糊时 | 聚焦核心价值，帮 research 缩小范围 |
| **你觉得最难的地方在哪？** | 用户有一定思考时 | 提前识别风险，research 可以针对性调研 |

### Step 5: 生成 01-specify.md

将对话中获取的信息整理为结构化文档。

```markdown
# <功能名> — 需求规格

## 基本信息
- **文档类型**: <PRD/功能设计/提案/完整产品>
- **复杂度**: <small/medium/large>
- **创建日期**: <YYYY-MM-DD>

## 动机
- **为什么做**: <一句话总结动机>
- **动机类型**: <自己有痛点/市场需求/公司要求/灵感兴趣>
- **背景故事**: <动机对话中的关键细节>

## 问题描述
<用户确认后的问题/痛点描述>

## 产品形态
<App/Web/CLI/硬件/...>

## 已知约束
- <约束 1>
- <约束 2>（如无则写"无已知约束"）

## 现有替代方案
<用户现在怎么解决的，或"无替代方案">

## 灵魂追问记录
- **成功标准**: <做成什么样算成功>
- **<其他被回答的灵魂问题>**: <答案>

## 调研关键词
<从以上内容中提炼，供 research 阶段使用>
<包含：问题相关词、产品形态词、现有替代方案名、用户提到的关键概念>

## 开放问题
- <需要 research 阶段回答的问题>
- <对话中暴露的不确定性>
- <用户自己也不确定的点>

## 待确认事项（需要在 value/plan 阶段定的）
- 产品方向（需 research 数据 + value 讨论后确认）
- 功能范围（需 plan 阶段定义）
- 目标用户画像（需 research 数据支撑后细化）
- 验收标准（需 value 讨论后在 plan 定义）
```

## 交接

本阶段完成后：
1. 将产出写入 `.build/01-specify.md`
2. 更新 `_manifest.json`：`specify.status = "completed"`，`specify.output = "01-specify.md"`，`specify.completed_at = "<时间>"`
3. 向用户展示一句话摘要："需求规格已完成，核心问题是 [问题描述的精简版]"
4. 交回编排器控制权（编排器会询问用户是否执行下一阶段）
