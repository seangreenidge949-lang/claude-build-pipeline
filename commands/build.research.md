# Stage 2: 调研 (Research)

## Stage Contract
- **requires**: `.build/01-specify.md`
- **optional**: —
- **produces**: `.build/02-research.md`
- **precondition**: specify 阶段已 completed

## 前置检查

运行前置检查脚本：
```bash
~/.claude/scripts/check-build-prerequisites.sh research --json
```

解析 JSON 输出。如果脚本返回非零退出码（specify 未完成），停止执行并告知用户。
从 `available_docs` 中确认 `01-specify.md` 存在，读取它获取调研锚点和关键词。

## 必读经验

进入本阶段前，读取 `~/.claude/memory/details/research/` 下所有经验文件：
- `~/.claude/memory/details/research/search-strategy.md` — 搜索工具优先级
- `~/.claude/memory/details/research/scrapling-tips.md` — 搜索体系架构和平台经验

## 执行流程

> **⛔ Rationalization → Reality**
> | 借口 | 现实 |
> |------|------|
> | "需求已经很清楚了，不需要调研" | 用户说的清楚 ≠ 内部已有决策/文档不存在冲突 |
> | "搜了没找到，调研结束" | 换关键词、换平台、换搜索方式再试 |

### Step 1: 读取调研输入

从 `01-specify.md` 提取：
- **动机和问题描述** → 搜索关键词的核心来源
- **产品形态** → 决定搜什么品类的竞品
- **现有替代方案** → 竞品搜索的起点
- **调研关键词** → 直接用于搜索查询
- **开放问题** → 调研需要回答的问题

### Step 2: 确定调研策略

从 `_manifest.json` 读取 `complexity`，决定调研深度和执行方式：

| 调研维度 | medium（主 agent 执行） | large（spawn subagent） |
|---------|----------------------|----------------------|
| 飞书内部 | 主 agent `search-doc` 1-2 次 | spawn `feishu-researcher` |
| 用户心声 | 主 agent WebSearch 2-3 条查询 | spawn `web-collector`（模板 B） |
| 竞品拆解 | 主 agent 搜 1 个竞品 | spawn `web-collector`（模板 C） |

> large 时三路并行 spawn，返回后由主 agent 直接汇总（不再 spawn info-analyst 二次汇总，减少一层 token 消耗）。

**subagent spawn 注意事项**：
- 🔒 spawn subagent 时必须指定 `mode: "bypassPermissions"`，否则 subagent 无法执行 Bash 命令（search.py 需要通过 Bash 调用）
- 如果 subagent 因权限问题失败，主 agent 自己用 WebSearch + search.py 执行调研，不再重试 spawn

**搜索工具选择策略**：
- **中文互联网内容**（用户心声、中文竞品、国内平台评论）→ 优先使用 `/web-research` skill（底层走 search.py，覆盖 16+ 中文平台）
- **英文/全球内容**（国际竞品、英文评测、技术文档）→ 用 WebSearch
- **WebSearch 返回 0 结果时** → 立即切换到 `/web-research` skill 重试，不要换关键词后继续用 WebSearch
- **需要深入提取某个页面内容时** → WebFetch 或 `/scrape`（反爬站点）

**搜索策略——确保有效结果**：
- 每个搜索查询同时准备中文和英文版本
- 关键词组合策略：`[核心概念] + [修饰词]`，修饰词轮换（痛点/推荐/评测/对比/替代/教程）
- 单次搜索 0 结果时：先换工具（WebSearch ↔ web-research），再换关键词组合，至少 3 种组合后才标记"未找到"
- 发现有价值的具体链接后用 WebFetch 深入提取

### Step 3: 执行调研

#### 3.1 飞书内部调研

**medium**：直接用 `search-doc` + `fetch-doc` 搜 1-2 个关键词。

**large**：spawn `feishu-researcher` subagent（model: sonnet），使用以下 prompt 模板：

```
调研目标：在飞书文档中找到与"[问题陈述]"相关的已有决策、方案、数据。

搜索策略：
1. 用以下关键词组合搜索（每组至少搜 1 次）：
   - [调研关键词列表，来自 01-specify.md]
   - [目标用户] + [问题相关词]
2. 找到文档后，提取：已有决策、数据指标、相关负责人

输出格式（严格遵守）：
## 飞书调研结果
### 关键文档（按相关性排序）
1. [文档标题] — [一句话摘要]（链接：[URL]）
   - 关键信息：[与本项目直接相关的内容]
### 已有决策
- [决策内容]（来源：[文档名]）
### 冲突/矛盾
- [如有不同文档说法不一致的情况]
### 信息缺口
- [调研未能覆盖的问题]

质量门槛：
- 只返回与问题陈述直接相关的文档，不返回"沾边"的
- 每个关键信息必须标注来源文档
```

#### 3.2 互联网用户心声

**medium**：主 agent 用 WebSearch 搜索 2-3 条查询，从 01-specify.md 的调研关键词构造：
- `{目标用户} {问题描述} 痛点/吐槽/推荐`
- `{现有替代方案名} 评测/对比/缺点`

**large**：spawn `web-collector` subagent（model: sonnet），使用以下 prompt 模板：

```
调研目标：收集真实用户对"[问题/领域]"的评价、痛点和需求。

搜索策略：
1. 搜索查询（按优先级）：
   - "[目标用户] [问题描述] 痛点/吐槽/推荐"
   - "[现有替代方案名] 评测/对比/缺点"
   - "[领域] 用户需求/市场"
2. 平台选择：[根据产品形态选择]
   - 硬件/数码 → 酷安、什么值得买、中关村在线、京东评论
   - App/软件 → 应用商店评论、小红书、知乎
   - 通用/B端 → 知乎、微博、行业论坛
3. 每个查询至少获取 3 条有效结果

输出格式（严格遵守）：
## 用户心声
### Top 痛点（按出现频率排序）
1. [痛点描述]（出现 N 次）
   - 原声："[用户原话]"（来源：[平台+链接]）
   - 原声："[用户原话]"（来源：[平台+链接]）
2. ...
### 用户好评点
- [好评内容]（来源：[平台+链接]）
### 未被满足的需求
- [需求描述]（来源：[平台+链接]）

质量门槛：
- 痛点必须有 ≥2 条不同来源的原声佐证
- 不返回明显营销/水军内容
- 每条原声必须标注平台和链接
```

#### 3.3 竞品功能拆解

**medium**：主 agent 搜索 1 个竞品（从 01-specify.md 的"现有替代方案"出发），提取核心功能和用户评价。

**large**：spawn `web-collector` subagent（model: sonnet），使用以下 prompt 模板：

```
调研目标：找出"[产品品类]"的 Top 3 竞品，拆解功能和商业模式。

搜索策略：
1. 搜索 "[品类] 推荐/排行/对比" 找到竞品列表
2. 从 "[现有替代方案名]" 出发扩展相关竞品
3. 对每个竞品搜索 "[竞品名] 功能介绍/评测/定价"
4. 优先从官网、应用商店描述、权威评测获取功能列表

输出格式（严格遵守）：
## 竞品拆解
### 竞品 1: [产品名]
- **一句话定位**：
- **核心功能**：（功能列表，来源：[官网/应用商店]）
- **定价模式**：免费/付费/订阅，价格区间
- **用户规模**：（如能找到）
- **优势**：（附用户评价原文 + 来源）
- **劣势**：（附用户吐槽原文 + 来源）
- **差异化机会**：[与本项目的问题陈述对比，这个竞品没做好或没做的地方]

### 竞品 2: ...
### 竞品 3: ...

## 竞品对比矩阵
| 维度 | 竞品1 | 竞品2 | 竞品3 | 本项目机会 |
|------|-------|-------|-------|-----------|
| ... | ... | ... | ... | ... |

质量门槛：
- 功能列表必须来自官网或应用商店，不能靠猜
- 定价信息必须标注来源和查询日期
- 如果找不到 3 个竞品，返回找到的 + 说明搜索过程
```

### Step 4: 调研覆盖度检查

| complexity | 飞书源最少 | 互联网源最少 | 用户心声最少 | 竞品最少 |
|-----------|-----------|------------|------------|---------|
| medium | 1 篇 | 3 条查询 | 3 条原声 | 1 个 |
| large | 3 篇 | 8 条查询 | 8 条原声 | 3 个 |

未达到最低覆盖度时，补充搜索。如果确实搜不到更多，标注"已穷尽可用源"并说明原因。

### Step 5: 整合调研结果 + 回答开放问题

将所有调研结果整合，**特别关注 01-specify.md 中的"开放问题"列表**——逐项标注是否已被调研回答。

写入 `.build/02-research.md`：

```markdown
# <功能名> — 调研报告

## 调研概况
- **调研深度**: <medium/large>
- **调研日期**: <YYYY-MM-DD>
- **调研维度**: 飞书内部 ✅/❌ | 用户心声 ✅/❌ | 竞品拆解 ✅/❌

## 飞书内部发现
### 关键文档
- <文档标题> — <一句话摘要>（来源：<飞书链接>）

### 已有决策
- <此前已做出的相关决策，需遵守>

### 冲突信息
- <如果发现内部文档之间有矛盾>

## 用户心声分析
### Top 痛点
1. <痛点 1>（来源数：N，典型原声："..."）
2. <痛点 2>
3. <痛点 3>

### 用户好评点
- <用户认可的方面>

### 未被满足的需求
- <需求描述>

## 竞品拆解
### 竞品概览
| 竞品 | 定位 | 核心差异 | 定价 |
|------|------|---------|------|
| ... | ... | ... | ... |

### 竞品对比矩阵
| 维度 | 竞品1 | 竞品2 | 竞品3 | 本项目机会 |
|------|-------|-------|-------|-----------|
| ... | ... | ... | ... | ... |

### 差异化机会总结
- <基于竞品分析，本项目可以切入的独特角度>

## 开放问题回答
| 问题（来自 01-specify.md） | 调研结论 | 置信度 |
|---------------------------|---------|--------|
| <问题 1> | <结论> | 高/中/低 |
| <问题 2> | <未能回答，原因：...> | — |

## 信息缺口
- <调研未能覆盖、需要后续确认的信息>

## 对后续阶段的建议
- **对价值判断的建议**: <哪些痛点最有说服力，质疑应该聚焦哪里>
- **对方案设计的建议**: <哪些已有决策需要遵守，竞品的哪些做法值得借鉴>
- **风险提示**: <调研中发现的潜在风险>
```

### Step 6: 信息可信度标注

- **飞书来源**：视为可信源，标注出处即可
- **互联网来源**：尽量两个不关联信息互证；无法满足时标注 `[单一来源]`
- **无佐证数据**：不使用，标记为 `[待补充数据]`

## 交接（循环式确认）

本阶段完成后，进入**调研展示→追问→补充**循环：

### 循环 Step A: 展示调研结果 + 建议补充方向

向用户展示调研摘要，并主动建议可能值得深入的方向：

```
📊 调研完成。以下是关键发现：

**用户痛点 Top 3**：
1. [痛点] — [原声引用]
2. ...

**竞品格局**：
- [竞品1]：[一句话定位]，[核心差异]
- [竞品2]：...

**差异化机会**：
- [基于调研发现的切入点建议]

📌 调研新发现（specify 中未提及）：
1. [发现描述] — 可能影响：[对产品方向的影响]

💡 建议补充调研的方向：
- [方向 A]：因为 [原因]
- [方向 B]：因为 [原因]

👉 你想：
1. 深入了解某个方向（告诉我哪个）
2. 调研其他你关心的方面
3. 调研足够了，继续下一步
```

### 循环 Step B: 补充调研

如果用户选择深入或补充：
1. 根据用户指定的方向执行补充搜索
2. 将新结果追加到 02-research.md
3. 回到 Step A，再次展示更新后的结果 + 新的补充建议

### 循环 Step C: 确认完成

用户确认调研足够后：
1. 将最终产出写入 `.build/02-research.md`
2. 更新 `_manifest.json`：`research.status = "completed"`，`research.output = "02-research.md"`，`research.completed_at = "<时间>"`
3. 如果调研发现影响了 01-specify.md 的问题描述或开放问题，更新 01-specify.md
4. 交回编排器控制权（编排器会询问用户是否执行下一阶段）
