# Stage 3: 价值判断与方向共创 (Value)

## Stage Contract
- **requires**: `.build/01-specify.md`
- **optional**: `.build/02-research.md`
- **produces**: `.build/03-value.md`
- **precondition**: specify 阶段已 completed

## 前置检查

运行前置检查脚本：
```bash
~/.claude/scripts/check-build-prerequisites.sh value --json
```

解析 JSON 输出。如果脚本返回非零退出码（specify 未完成），停止执行并告知用户。
从 `available_docs` 读取可用的前序文件：`01-specify.md`（必读）、`02-research.md`（如有）。

## 本阶段的定位

> **value 不是冷面质疑官，而是帮用户在数据基础上想清楚方向的伙伴。**
>
> 质疑是手段，不是目的。这个阶段的终极产出是：一个**经过讨论确认的产品方向**——用户看完调研、讨论过价值、回答了关键问题后，清晰地知道要做什么方向、为什么选这个方向。
>
> 方向在 specify 阶段是定不了的（没有数据），在 plan 阶段又太晚了（已经开始写方案）。
> value 阶段是唯一合适的方向决策点。

## 执行流程

### Step 0: 复杂度检查

从 `_manifest.json` 读取 `complexity` 字段：
- **small**: 本阶段由编排器根据用户选择决定是否执行。如果进入，走 quick 模式
- **medium**: 进入简化讨论模式（Step 1-quick）
- **large**: 进入完整讨论模式（Step 1-5）

---

### 简化讨论模式（small/medium 复杂度）

#### Step 1-quick: 快速方向确认

读取 `01-specify.md` 和 `02-research.md`（如有），用 AskUserQuestion 一次性完成：

```
基于你的问题和调研发现，我有几个关键判断想和你确认：

1. [基于调研的核心发现/质疑]
   - 选项 A / 选项 B / 你有其他想法

2. 产品方向建议：
   - 方向 A（推荐）：[基于调研数据的理由]
   - 方向 B：[理由]

3. 你觉得这个方向的成功概率有多大？
   - 很有信心 / 有一定风险但值得试 / 不太确定
```

用户回答后，直接产出 03-value.md（简洁版），跳转到交接。

---

### 完整讨论模式（large 复杂度）

#### Step 1: 阅读并整合背景

读取 `01-specify.md`（动机、问题、灵魂追问记录）和 `02-research.md`（调研发现），整合出：
- 用户的核心问题和动机
- 调研揭示的市场真相（痛点、竞品、差异化机会）
- 两者之间的**差距或矛盾**（用户以为的 vs 调研发现的）

#### Step 2: 基于调研的深度讨论（逐轮推进）

> 不是预设 6 个维度逐个质疑，而是**根据 research 的实际发现动态构建讨论话题**。

<HARD-GATE>
一次只讨论一个话题。每个话题要从调研数据出发，不是空对空的质疑。
讨论是有建设性的——每提出一个问题/挑战，同时给出你基于调研的建议。
目标不是否定用户的想法，而是帮他/她看到更完整的图景。
</HARD-GATE>

**讨论话题的选择**（根据 research 发现动态决定优先级）：

| 调研发现 | 讨论重点 | 讨论方式 |
|---------|---------|---------|
| 竞品已很成熟 | 差异化在哪？为什么用户会选你而不是现有方案？ | 展示竞品矩阵 + 讨论空白点 |
| 用户痛点和预期不完全匹配 | 真正的痛点是什么？你解决的是最痛的还是最容易的？ | 列出痛点排序 + 讨论优先级 |
| 市场有明显空白 | 为什么还没人做？是真机会还是伪需求？ | 探讨可能的原因 |
| 内部已有相关决策/尝试 | 之前为什么没做成？这次有什么不同？ | 引用飞书调研发现 |
| 调研数据不足 | 哪些关键假设还没验证？风险有多大？ | 诚实标注不确定性 |

**对话规则**：
- 每轮提出一个话题 + 你的观点/建议，等用户回应
- 用户的回答如果能说服你（有数据/逻辑），接受并推进
- 用户不确定时，给出 2-3 个选项帮他选，而不是追着问
- 全部话题讨论完（通常 3-5 轮）进入 Step 3

#### Step 3: 方向提议

基于讨论结果，**主动提出 2-3 个产品方向**（不是等用户想）：

```
基于我们的讨论和调研数据，我建议以下方向：

**方向 A：[名称]**（推荐）
- 切入点：[具体做什么]
- 为什么选这个：[调研数据 + 讨论中的共识]
- 风险：[坦诚标注]
- 预期成功标准：[基于灵魂追问的成功标准细化]

**方向 B：[名称]**
- 切入点：...
- 为什么也值得考虑：...
- 风险：...

**方向 C：[名称]**（如适用）
- ...
```

使用 AskUserQuestion 让用户选择方向。用户也可以提出自己的方向。

#### Step 4: 开放问题收口

读取 `01-specify.md` 的"开放问题"和"待确认事项"列表，逐项检查：

| 问题 | 状态 | 来源 |
|------|------|------|
| <问题 1> | ✅ 已回答（research 数据 / value 讨论） | 置信度：高 |
| <问题 2> | ⚠️ 部分回答，置信度 < 90% | 需要问用户确认 |
| <问题 3> | ❌ 未回答 | 需要问用户 |

对置信度 < 90% 的问题和未回答的问题，直接用 AskUserQuestion 向用户确认。

#### Step 5: 生成 03-value.md

```markdown
# <功能名> — 价值判断与方向确认

## 讨论结论
- **确认的产品方向**: <用户选定的方向>
- **为什么选这个方向**: <调研数据 + 讨论共识>
- **把握度**: X/10
- **讨论日期**: <YYYY-MM-DD>

## 核心论据（支持这个方向的最强理由）
1. <论据 1>（来源：调研/讨论）
2. <论据 2>
3. <论据 3>

## 主要风险
- <风险 1>：<影响> / <应对思路>
- <风险 2>：<影响> / <应对思路>

## 差异化定位
- **与竞品 A 的差异**: <...>
- **核心竞争力**: <...>

## 开放问题收口
| 问题 | 结论 | 来源 |
|------|------|------|
| ... | ... | research/讨论/用户确认 |

## 方向探索记录
### 讨论了哪些方向
- 方向 A（选定）：<理由>
- 方向 B（放弃）：<为什么>
- 方向 C（放弃）：<为什么>

### 关键讨论摘要
- [话题 1]：用户观点 → AI 建议 → 共识
- [话题 2]：...

## 对 plan 阶段的输入
- **产品方向**: <一句话>
- **核心切入点**: <具体做什么>
- **成功标准**: <细化后的标准>
- **优先级建议**: <基于讨论的功能优先级>
- **需要在 plan 中特别注意的**: <...>
```

## 交接

本阶段完成后：
1. 将产出写入 `.build/03-value.md`
2. 更新 `_manifest.json`：`value.status = "completed"`，`value.output = "03-value.md"`，`value.completed_at = "<时间>"`
3. 向用户展示一句话摘要："方向已确认：[产品方向]，核心切入点是 [切入点]"
4. 交回编排器控制权（编排器会询问用户是否执行下一阶段）
