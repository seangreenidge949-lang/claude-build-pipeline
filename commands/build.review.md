# Stage 9: 验收检查 (Review)

## Stage Contract
- **requires**: `.build/01-specify.md`
- **optional**: 所有已完成阶段的产出文件
- **produces**: `.build/09-review.md`
- **precondition**: specify 阶段已 completed，至少有一个后续阶段已 completed

## 前置检查

运行前置检查脚本：
```bash
~/.claude/scripts/check-build-prerequisites.sh review --json
```

解析 JSON 输出。如果脚本返回非零退出码（specify 未完成），停止执行并告知用户。
从 `available_docs` 读取所有已存在的前序产出文件，用于全面验收。

## 必读经验

进入本阶段前，读取 `~/.claude/memory/details/review/` 下所有经验文件：
- `~/.claude/memory/details/review/user-perspective-validation.md` — 用户视角验证 checklist
## 执行流程

### Step 1: 确定验收范围

根据 manifest 中已完成的阶段，确定验收内容：

| 已完成到 | 验收重点 |
|---------|---------|
| plan | 文档完整性、逻辑一致性、可执行性 |
| design | 上述 + 设计稿与方案一致性 |
| implement | 上述 + 代码功能覆盖率、测试通过率 |

### Step 2: 文档验收（如果有 plan）

> **TaskCreate 追踪**：为以下每一项创建任务，逐项验收并标记完成。未通过的项记录到验收报告的"问题"章节。

- [ ] 问题陈述清晰且有数据支撑
- [ ] 目标可量化（非模糊描述）
- [ ] 范围边界明确（包含"不做什么"）
- [ ] 边界情况已覆盖
- [ ] 依赖项已列出
- [ ] 全文一致性：开关默认值、功能命名等关键设定在所有章节一致
- [ ] 数据映射完整：所有"动态文案"、"等级判断"均有完整映射表
- [ ] 技术依赖已确认
- [ ] 开发可实现性：API/数据源已定义、交互可编码、逻辑无歧义

### Step 3: 设计验收（如果有 design）

> **TaskCreate 追踪**：为以下每一项创建任务，逐项验收并标记完成。

- [ ] 设计稿覆盖方案中所有页面
- [ ] 交互逻辑与方案描述一致
- [ ] 空态、加载态、错误态已设计
- [ ] 视觉规格可编码（HEX、具体数值）
- [ ] UI 还原度（如有实现）：实际界面截图与 `.build/screenshots/` 下的设计稿 PNG 逐页对比，布局/间距/颜色/文案一致

### Step 4: 代码验收（如果有 implement）

> ⚠️ **审查原则**：假设 implement-log 中的 ✅ 状态过于乐观。必须独立运行验证命令确认。代码中的 TODO、FIXME、hack 可能被遗漏在日志中。

> **TaskCreate 追踪**：为以下每一项创建任务，逐项验收并标记完成。

- [ ] 所有任务标记为完成
- [ ] 项目可正常启动/运行
- [ ] 核心功能可正常使用
- [ ] 测试通过（如有）
- [ ] 无明显安全漏洞
- [ ] 代码结构与 code-plan 一致

### Step 5: 需求覆盖率检查

对照 `01-specify.md` 中的需求逐项核对：

```markdown
| 需求项 | 覆盖状态 | 验证方式 | 备注 |
|--------|---------|---------|------|
| <需求 1> | ✅ 已覆盖 | <方案第X节/代码文件> | — |
| <需求 2> | ⚠️ 部分覆盖 | <...> | <缺少什么> |
| <需求 3> | ❌ 未覆盖 | — | <原因> |
```

**审查迭代上限**：最多 3 轮。如果 3 轮后仍有未解决的问题，列出遗留事项交由用户决定如何处理，不再继续审查循环。

### Step 6: 生成验收报告

写入 `.build/09-review.md`：

```markdown
# <功能名> — 验收报告

## 验收概况
- **验收日期**: <YYYY-MM-DD>
- **验收范围**: <文档/设计/代码>
- **总体结论**: <通过/有条件通过/未通过>

## 需求覆盖率
- 总需求项: <N>
- 已覆盖: <X>（XX%）
- 部分覆盖: <Y>
- 未覆盖: <Z>

## 验收明细
### 文档验收
<checklist 结果>

### 设计验收（如适用）
<checklist 结果>

### 代码验收（如适用）
<checklist 结果>

## 发现的问题
| # | 严重程度 | 描述 | 建议处理 |
|---|---------|------|---------|
| 1 | 高/中/低 | <...> | <...> |

## 遗留事项
- <待后续版本处理的问题>

## 改进建议
- <对流程/方案/实现的改进建议>
```

### Step 7: 迭代决策

验收完成后，根据验收结果使用 AskUserQuestion 询问用户下一步：

**情况 A：全部通过（需求覆盖率 100%，无高/中严重问题）**

选项：
- 结束（推荐）— 项目已满足所有验收标准，生成交付摘要
- 优化迭代 — 基于以下改进建议继续优化：[列出 09-review.md 中的改进建议]
- 新增功能 — 在现有基础上扩展新功能（回到 specify）

**情况 B：有未通过项**

先分析每个未通过项的根因，推荐回到对应阶段：

| 问题类型 | 推荐回到 |
|---------|---------|
| 需求描述不清/有歧义 | specify |
| 方案逻辑缺陷/功能遗漏 | plan |
| 设计还原度差 | design 或 implement |
| 代码 bug / 功能不正确 | implement |
| 部署/启动问题 | deploy |

选项：
- 回到 <推荐阶段> 修复（推荐）— [列出需要修复的问题]
- 回到其他阶段 — [列出可选阶段]
- 记录为遗留事项，结束本轮

**回环执行机制**：

用户选择回到某阶段后：
1. 将该阶段及之后所有阶段的 status 重置为 `pending`
2. manifest 的 `iteration` 字段 +1
3. `iteration_history` 追加本轮信息：
   ```json
   {
     "round": <当前轮次>,
     "completed_at": "<ISO 8601>",
     "review_result": "<all_passed | issues_found>",
     "issues_count": <N>,
     "restart_from": "<阶段名>"
   }
   ```
4. 上一轮的产出文件不删除（回环阶段会读取并覆盖写入）
5. 交回编排器，从选定阶段重新开始执行

## 交接

本阶段完成后：
1. 将产出写入 `.build/09-review.md`
2. 更新 `_manifest.json`：`review.status = "completed"`，`review.output = "09-review.md"`，`review.completed_at = "<时间>"`
3. 向用户展示验收结论
4. 生成最终交付摘要（由编排器 build.md 负责）
5. 交回编排器控制权
6. 如果用户在 Step 7 选择了回环，则不生成交付摘要，而是交回编排器执行回环逻辑
